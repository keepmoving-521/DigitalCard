# V1.0 发布、回滚与缺陷响应

## 发布门禁

发布候选版本必须通过代码规范、后端测试、前端类型检查、组件测试、前端构建、迁移升级、备份恢复演练、安全检查、性能基线和 MVP 主流程回归。版本号、README、CHANGELOG 和 `docs` 必须一致。

## 发布步骤

1. 冻结候选提交并记录迁移头；
2. 创建发布前备份并验证校验清单；
3. 构建不可变镜像并在隔离环境升级；
4. 执行创建企业、员工录入、名片发布、公开访问、留资、领取、转客户和跟进冒烟测试；
5. 部署生产并观察健康、错误率和 P95；
6. 记录发布时间、版本、负责人和已知问题。

## 回滚决策

出现登录不可用、严重跨租户访问、数据写入损坏、公开主链路中断或持续高 5xx 时立即停止发布。优先回滚应用；数据库迁移只有在确认旧应用不兼容且已停止写入后才回退。恢复备份前必须保存故障现场副本。

## 缺陷级别

| 级别 | 示例 | 响应目标 |
| --- | --- | --- |
| P0 | 跨租户泄露、数据大面积损坏、系统完全不可用 | 立即响应，停止服务或隔离风险 |
| P1 | 登录、名片发布、公开留资或跟进主链路阻断 | 2 小时内响应并提供缓解方案 |
| P2 | 单个功能异常但有替代路径 | 1 个工作日内确认计划 |
| P3 | 体验、文案和低风险问题 | 纳入正常迭代 |

修复完成后必须补自动化回归、记录根因和影响范围，并确认文档、监控或操作流程是否需要更新。
