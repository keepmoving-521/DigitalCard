# DigitalCard 版本迭代需求文档

> 文档状态：持续维护
>
> 适用阶段：项目初始化至 V1.5
>
> 更新日期：2026-06-30

## 1. 文档目的

本文档将 DigitalCard 的产品愿景拆解为可开发、可测试、可验收的小版本，用于产品排期、技术设计、测试验收和发布复盘。

版本采用 `主版本.能力版本.修订版本` 规则：

- `V0.x`：完成工程底座和数字名片 MVP；
- `V1.0`：形成可对外试用的首个稳定版本；
- `V1.x`：补齐数据经营、营销、SaaS 与 AI 能力；
- 修订版本仅处理缺陷、安全更新和小范围体验优化，不增加大型业务模块。

建议每个小版本以 1～2 周为一个迭代周期。实际周期应根据团队规模、原型完成度和外部依赖调整。

## 2. 产品目标与边界

### 2.1 产品目标

1. 企业能够统一维护组织、员工、品牌信息和数字名片。
2. 员工能够快速生成、预览和分享个人名片。
3. 客户能够浏览企业、员工和产品信息，并完成咨询或留资。
4. 企业能够沉淀客户行为、开展跟进并衡量转化效果。
5. 平台后续能够以多租户 SaaS 方式运营，并接入 AI 与开放能力。

### 2.2 核心角色

| 角色 | 权限边界 |
| --- | --- |
| 平台运营员 | 管理租户、套餐、平台配置和运行状态，不默认查看租户业务数据 |
| 企业管理员 | 管理本企业资料、部门、员工、角色、内容和数据报表 |
| 普通员工 / 销售 | 维护授权范围内的个人名片、客户和跟进记录 |
| 访客 / 客户 | 无需登录即可浏览公开内容，经明确授权后提交个人信息 |

### 2.3 首阶段不包含

- 原生 iOS、Android App 和微信小程序；
- 在线支付、电子合同及复杂订单系统；
- 自研即时通信和音视频通话；
- 大规模微服务拆分、多地域容灾；
- 自动替代销售决策的全自动 AI 工作流。

## 3. 全局需求

以下约束从首个业务版本开始持续生效：

- 所有租户业务数据必须包含明确的租户归属，服务端不得仅依赖前端参数进行数据隔离；
- 密码不得明文保存，登录和敏感操作应具备防暴力尝试能力；
- 手机号、邮箱、客户备注等敏感信息按权限展示，并记录关键操作日志；
- 公开页面不得暴露内部编号、后台字段或未发布内容；
- 删除重要数据优先采用可恢复的逻辑删除，并明确恢复和彻底删除策略；
- 用户上传文件必须校验类型、大小和访问权限；
- 所有时间统一存储，展示时按用户或企业时区转换；
- 涉及客户留资和数据分析时，应提供隐私说明、授权提示与必要的退出机制。

## 4. 小版本迭代规划

### V0.1.0：工程底座（已实现）

**版本目标**：建立可持续开发、测试和部署的基础工程。

**需求范围**：

- 初始化 FastAPI 后端、Vue 3 管理端和基础 H5 工程；
- 建立开发、测试、生产环境配置规范及 `.env.example`；
- 接入数据库迁移、统一错误响应、请求日志和健康检查；
- 提供本地一键启动方式与最小容器配置；
- 建立代码格式检查、单元测试和持续集成流程；
- 自动生成并访问 API 文档。

**验收标准**：

- 新成员按 README 能在 30 分钟内启动项目；
- 健康检查、前端首页和 API 文档均可访问；
- 提交代码时自动执行格式检查和测试；
- 敏感配置不进入版本库，应用启动时能提示缺失配置。

**暂不包含**：业务页面、用户注册和正式部署集群。

### V0.2.0：账户、登录与基础安全（已实现）

**版本目标**：支持后台用户安全登录并形成最小账户体系。

**需求范围**：

- 账号密码登录、退出、刷新登录状态；
- 当前用户资料查询和密码修改；
- 管理员创建账户、启用、停用和重置密码；
- 登录失败限制、密码强度规则和登录日志；
- 前端登录页、鉴权路由和无权限页面。

**验收标准**：

- 有效账户可登录并访问授权页面，停用账户立即失去访问权限；
- 未登录、登录过期和权限不足返回不同且统一的错误；
- 修改或重置密码后，旧凭据按安全策略失效；
- 日志中不得记录明文密码、令牌等敏感信息。

**暂不包含**：短信登录、第三方 OAuth、复杂单点登录。

### V0.3.0：企业、租户隔离与权限（已实现）

**版本目标**：建立企业空间、组织架构和可执行的权限边界。

**需求范围**：

- 创建和维护企业名称、Logo、简介、联系方式与状态；
- 部门树的新增、编辑、排序、移动和停用；
- 预置企业管理员、内容管理员、销售、普通员工角色；
- 支持角色与权限点绑定，后端接口执行权限校验；
- 所有企业数据自动限定在当前租户范围；
- 记录企业资料、角色和权限的变更日志。

**验收标准**：

- 不同企业使用相同业务编号时仍无法互相读取或修改数据；
- 无相应权限的用户看不到入口，直接请求接口同样被拒绝；
- 停用部门前能识别并提示仍在部门内的员工；
- 企业管理员不能操作平台级配置。

**暂不包含**：集团多企业切换、自定义数据权限表达式。

### V0.4.0：员工管理

**版本目标**：让企业完成员工档案维护和账号关联。

**需求范围**：

- 员工新增、编辑、查询、分页、筛选、停用和恢复；
- 员工与部门、职位、直属上级、登录账户关联；
- 支持头像、姓名、手机号、邮箱、个人简介等字段；
- 批量导入员工并返回逐行校验结果；
- 管理员发送初始账户邀请或重新邀请；
- 员工仅能编辑企业允许自行维护的个人字段。

**验收标准**：

- 员工编号、手机号或邮箱的唯一规则有明确校验；
- 导入文件中部分数据错误时，不影响错误定位和正确数据处理；
- 停用员工无法登录，公开名片按企业策略隐藏或显示离职状态；
- 员工不能修改自己的角色、部门等受控字段。

**暂不包含**：考勤、薪资、人事审批等 HR 系统能力。

### V0.5.0：名片资料与模板

**版本目标**：企业和员工能够配置、预览并发布数字名片。

**需求范围**：

- 名片基础资料、联系方式、社交账号和展示模块配置；
- 企业管理员创建企业默认模板，配置主题色、Logo 和模块顺序；
- 员工基于模板维护允许个性化的内容；
- 草稿、已发布、已下线三种名片状态；
- 后台实时预览桌面和移动端效果；
- 发布前执行必填项、图片和公开信息校验。

**验收标准**：

- 未发布的修改不会影响线上名片；
- 企业锁定的品牌字段不可被员工覆盖；
- 名片发布、下线与再次发布状态流转正确；
- 无权限用户不能编辑他人名片。

**暂不包含**：自由拖拽页面设计器、付费模板商城、NFC 写卡。

### V0.6.0：H5 名片与分享闭环

**版本目标**：客户可稳定访问名片，员工可生成并追踪分享入口。

**需求范围**：

- 移动端优先的公开名片页和独立公开地址；
- 生成二维码、复制链接和带来源参数的分享链接；
- 一键拨号、发送邮件、复制微信号、保存通讯录；
- 展示企业信息及员工选择公开的内容；
- 记录页面访问、分享渠道、关键按钮点击等基础事件；
- 名片下线、员工停用和企业停用时展示合理状态页。

**验收标准**：

- 主流移动浏览器可正常访问，首屏和主要操作无明显布局错误；
- 二维码内容与对应名片、分享来源一致；
- 公开页只返回已发布且允许公开的数据；
- 同一访客的重复刷新不会造成明显异常统计膨胀。

**暂不包含**：微信开放平台深度能力、个性化短域名、NFC。

### V0.7.0：产品中心与素材库

**版本目标**：通过名片展示企业产品、案例和销售资料。

**需求范围**：

- 产品分类及产品的新增、编辑、排序、发布和下线；
- 产品封面、图集、介绍、规格、视频链接和附件；
- 企业素材库统一管理图片、视频和 PDF；
- 名片选择推荐产品，公开端查看产品列表和详情；
- 文件类型、大小、访问范围和引用关系校验。

**验收标准**：

- 下线产品和私有素材无法从公开地址访问；
- 删除被名片或产品引用的素材前给出明确提示；
- 产品排序及推荐关系在发布后正确生效；
- 上传失败可重试，且不会留下无业务归属的脏数据。

**暂不包含**：库存、价格促销、购物车和订单支付。

### V0.8.0：客户留资与线索分配

**版本目标**：把公开访问转化为可管理的销售线索。

**需求范围**：

- 名片咨询表单，支持姓名、联系方式、需求说明和隐私授权；
- 线索列表、详情、来源、归属员工和处理状态；
- 基于名片归属人的默认分配规则；
- 管理员转交线索，销售领取和标记无效；
- 相同联系方式的重复线索识别与提示；
- 新线索站内通知。

**验收标准**：

- 未同意隐私条款时不能提交个人信息；
- 线索记录能够追溯名片、产品、分享渠道和提交时间；
- 普通销售只能查看分配给自己或授权范围内的线索；
- 重复提交不会无提示地创建大量重复客户。

**暂不包含**：复杂线索评分、短信和企业微信通知。

### V0.9.0：客户档案与销售跟进

**版本目标**：形成从线索到客户、从跟进到商机的基础 CRM 闭环。

**需求范围**：

- 线索转为客户，维护客户联系人、标签和负责人；
- 跟进记录支持文本、跟进方式、时间和下次跟进提醒；
- 客户时间线统一展示来源、浏览、留资和跟进事件；
- 创建商机并维护预计金额、预计成交时间和阶段；
- 管理员配置基础商机阶段，查看简单漏斗；
- 客户转移、合并和归档。

**验收标准**：

- 线索转客户后来源信息和历史记录完整保留；
- 客户合并前预览冲突，合并后关联记录不丢失；
- 商机阶段变化被记录，可追溯操作人和时间；
- 客户负责人变更后，查看和编辑权限同步更新。

**暂不包含**：报价单、合同、回款、复杂销售预测。

### V1.0.0：MVP 稳定发布

**版本目标**：交付可供首批企业试用的稳定版本，验证“创建—分享—留资—跟进”主链路。

**需求范围**：

- 串联 V0.1～V0.9 的核心流程并修复阻断问题；
- 完成初始化向导、空状态、错误页和关键操作确认；
- 提供数据备份恢复、审计日志查询和基础运行监控；
- 补齐接口、部署、运维、隐私和用户使用文档；
- 执行安全检查、性能基线测试和核心流程回归；
- 建立版本发布、回滚和缺陷响应机制。

**验收标准**：

- 新企业可独立完成企业创建、员工录入、名片发布和分享；
- 客户留资后，销售可在后台完成领取、转客户和跟进；
- 核心业务接口达到团队约定的可用性和响应时间基线；
- 无阻断发布的高风险安全问题和严重数据隔离问题；
- 备份可在隔离环境完成恢复演练。

**发布指标**：名片发布成功率、公开页访问成功率、留资提交成功率、线索首次处理时长和系统错误率。

### V1.1.0：经营数据分析

**版本目标**：让企业知道名片从哪里被访问、哪些内容有效、是否产生转化。

**需求范围**：

- 统一访问、分享、点击、留资和转化事件模型；
- 企业、部门、员工、名片、产品和渠道维度看板；
- 趋势、排行、转化漏斗及时间范围筛选；
- 机器人流量、内部访问和重复事件的基础过滤；
- 报表导出及导出权限控制；
- 提供数据口径说明和最后更新时间。

**验收标准**：

- 看板数据可通过抽样事件追溯并核对；
- 不同角色只能查看授权组织或个人范围的数据；
- 指标名称、计算公式、去重规则在文档中明确；
- 大数据量查询不阻塞名片公开访问主链路。

### V1.2.0：营销表单与活动

**版本目标**：支持企业围绕名片和产品开展轻量营销获客。

**需求范围**：

- 可配置表单字段、必填规则、隐私说明和提交结果页；
- 创建活动、报名页、起止时间和人数限制；
- 表单或活动绑定名片、产品和分享渠道；
- 报名名单、状态管理、导出和线索转换；
- 防重复提交、频率限制和基础反垃圾策略；
- 活动效果统计。

**验收标准**：

- 活动未开始、已结束和名额已满时不能继续报名；
- 表单字段变更不破坏历史提交记录；
- 导出文件中的敏感信息符合角色权限；
- 每条报名可追溯活动、渠道和对应线索。

### V1.3.0：SaaS 运营与套餐配额

**版本目标**：让平台能够管理多个企业的生命周期和资源使用。

**需求范围**：

- 平台端租户查询、开通、冻结、恢复和注销流程；
- 套餐配置及员工数、名片数、存储量等配额；
- 租户试用期、到期状态和续期记录；
- 超限前提示、超限后限制新增但保护现有数据；
- 平台运营日志、租户使用概览和异常状态提醒；
- 平台运维临时协助租户时使用可审计的授权机制。

**验收标准**：

- 套餐变更后配额按明确规则即时或定时生效；
- 到期或冻结不会直接删除租户数据；
- 平台人员访问租户业务数据有授权、有期限、有日志；
- 租户注销包含确认、冷静期、导出和清理策略。

**暂不包含**：在线支付、自动开票、复杂渠道分佣。

### V1.4.0：企业知识库与 AI 助手

**版本目标**：使用企业已授权内容辅助客户问答和销售工作。

**需求范围**：

- 将企业介绍、产品、案例、FAQ 和已授权文档纳入知识库；
- 文档解析、索引、更新、停用和失败重试；
- 公开端 AI 问答，回答附带来源并支持转人工或留资；
- 后台生成企业介绍、产品文案和跟进建议草稿；
- 企业配置 AI 开关、使用范围、提示语和用量限制；
- 保存必要的调用记录、反馈和安全审计信息。

**验收标准**：

- AI 不得检索或引用其他租户的知识；
- 已下线或取消授权的内容在索引更新后不再用于回答；
- AI 生成内容明确标识为建议，需人工确认后才能发布；
- 无可靠依据时应明确表达不确定性并引导人工处理；
- 管理员可查看用量、失败率和用户反馈。

**暂不包含**：自动修改业务数据、自动承诺价格或合同条款。

### V1.5.0：消息中心与开放平台

**版本目标**：连接外部业务系统，并提高关键业务事件的触达效率。

**需求范围**：

- 站内消息、未读计数、已读和消息偏好；
- 新线索、待跟进、配额预警等消息模板；
- 开放 API 的应用创建、凭据轮换、权限范围和调用日志；
- 线索创建、客户更新等 Webhook 订阅；
- Webhook 签名、重试、幂等和失败记录；
- API 频率限制、版本管理和开发者文档。

**验收标准**：

- 应用凭据只在创建或轮换时完整展示，存储符合安全要求；
- API 无法跨越应用和租户的授权范围；
- Webhook 重试不会导致消费方重复创建业务数据；
- 调用方可查询失败原因，管理员可停用异常应用；
- 废弃接口有明确迁移周期和兼容策略。

## 6. 通用验收与完成定义

一个版本只有同时满足以下条件才能标记为完成：

- 产品需求和交互原型经过确认，范围变更有记录；
- 数据模型和接口变更包含迁移方案及必要文档；
- 核心业务逻辑具备自动化测试，关键用户流程通过回归测试；
- 租户隔离、权限、输入校验和敏感信息处理经过专项检查；
- 页面具备加载、空数据、失败、无权限等状态；
- 日志能够定位错误，但不包含密码、令牌和不必要的个人信息；
- 部署、配置和回滚方式已验证；
- README、API 文档和用户说明与当前版本一致；
- 已知问题已登记，不存在阻断发布的问题。

## 7. 非功能需求基线

### 安全与隐私

- 遵循最小权限原则，敏感操作需要再次确认；
- 登录凭据、应用密钥和隐私数据采用合适的加密或脱敏策略；
- 文件上传防止伪造类型、路径穿越和恶意文件直接执行；
- 对登录、公开表单、AI 问答和开放 API 设置频率限制；
- 建立客户数据导出、更正、注销和删除处理流程。

### 性能与可靠性

- 公开名片页面优先保障移动网络下的加载体验；
- 列表默认分页，导出、大文件处理和统计聚合使用异步任务；
- 对数据库、缓存、对象存储和外部 AI 服务设置超时与失败策略；
- 关键服务具备健康检查、错误监控、备份和恢复演练；
- 每次版本发布前记录性能基线，避免未经评估设定失真的绝对指标。

### 易用性与兼容性

- 管理端覆盖主流桌面浏览器当前稳定版本；
- H5 公开页覆盖主流移动浏览器，并适配常见屏幕宽度；
- 表单字段有清晰标签、校验提示和键盘操作支持；
- 关键操作结果可感知，危险操作可确认或撤销。

## 8. 版本评审建议

每次迭代开始前确认：业务价值、范围、原型、数据口径、技术依赖和验收负责人。每次发布后复盘：目标完成度、线上问题、核心指标变化、用户反馈和下一版本范围。

如果进度受限，应优先保证“企业建档—员工名片—公开分享—客户留资—销售跟进”主链路，AI、营销玩法和复杂平台运营能力不得反向阻塞 V1.0 MVP 发布。
