# V0.5.0 名片资料与模板说明

## 草稿与线上快照

每位员工在企业内只有一张数字名片。名片同时保存草稿数据和已发布快照：

- 保存草稿只增加草稿修订号，不改变当前状态和线上数据；
- 发布时合并企业模板与草稿，完成校验后生成完整、独立的公开快照；
- 发布后继续编辑会显示“有未发布修改”，公开接口仍返回上一次快照；
- 下线会保留快照，再次发布时使用当前草稿生成新快照。

这种设计确保企业模板更新、员工编辑中断或错误内容都不会静默影响线上名片。

## 企业默认模板

每个企业维护一个默认模板，包括模板名称、六位十六进制主题色、Logo、四个固定模块的展示顺序、品牌锁定字段和员工可编辑字段。

模块包括：个人信息、联系方式、社交账号和个人简介。当前版本支持排序，不提供自由拖拽布局。企业可以锁定主题色、Logo 和模块顺序；即使这些字段出现在员工可编辑列表中，锁定规则仍优先，员工接口会返回 `card_field_locked`。

企业管理员和内容管理员默认可以维护模板及代管员工名片。普通员工和销售只能编辑、预览、发布或下线自己的名片。所有他人名片接口都执行 `card.manage` 权限和租户范围校验。

## 发布校验

发布前服务端至少检查：

- 展示姓名不能为空；
- 电话或邮箱至少填写一项；
- 个人信息和联系方式模块必须存在；
- 头像与 Logo 地址必须是 HTTP 或 HTTPS 地址；
- 员工、企业均处于启用状态。

校验失败统一返回 `card_publish_validation_failed`，并在 `details` 中列出缺失字段和无效图片字段。

## 状态流转

```text
草稿 ──发布──> 已发布 ──下线──> 已下线
  ^              │                 │
  └──继续编辑────┴────继续编辑─────┘
                   已下线 ──再次发布──> 已发布
```

已发布状态下保存草稿不会退回草稿状态，系统通过修订号单独标记未发布修改。

## 主要接口

| 接口 | 用途 |
| --- | --- |
| `GET/PUT /api/v1/tenant/card-template` | 查看或维护企业默认模板 |
| `GET/PATCH /api/v1/tenant/cards/me` | 查看或编辑本人名片草稿 |
| `GET /api/v1/tenant/cards/me/preview` | 预览合并模板后的本人草稿 |
| `POST /api/v1/tenant/cards/me/publish` | 发布或再次发布本人名片 |
| `POST /api/v1/tenant/cards/me/offline` | 下线本人名片 |
| `GET/PATCH /api/v1/tenant/cards/{employee_id}` | 企业代管员工名片 |
| `GET /api/v1/tenant/cards/{employee_id}/preview` | 企业预览员工草稿 |
| `POST /api/v1/tenant/cards/{employee_id}/publish` | 企业发布员工名片 |
| `POST /api/v1/tenant/cards/{employee_id}/offline` | 企业下线员工名片 |
| `GET /api/v1/public/cards/{card_id}` | 获取当前已发布快照 |
